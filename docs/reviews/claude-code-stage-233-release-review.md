# Phase 3 Release Review (Stage 233)
**Verdict:** APPROVE_WITH_NITS

## Critical
None. No release-blocking defect. The collector is correct, fail-closed, bounded, and handles no passwords; the verified gate (pytest/ruff/hygiene/packaging/installed-wheel smoke) plus `uv.lock`/`pyproject.toml` unchanged all hold.

## Important
- **Design spec body contradicts the shipped implementation.** `docs/superpowers/specs/2026-06-30-phase3-instagram-design.md` is marked `Status: Complete — shipped as Stages 231-233`, but its Goal/Architecture/Credential/Tech-Stack sections still describe the *subprocess CLI* design that was explicitly rejected: "invokes it as a subprocess and parses its JSON metadata output", `instaloader <target> --metadata-json -D <tmp_dir>`, `shutil.which`, "instaloader binary path override", "stdlib `subprocess` ... shells out to it", `items_seen=<discovered count>`. The shipped code is library-based (`import instaloader`, `load_session_from_config`, lazy `islice` iteration, `items_seen=len(bounded)`). Comparing HEAD to the working tree, Stage 233 only bumped the Status line and left the body stale. This is **non-blocking** — the deviation is recorded in the Stage 232 plan (line 13), and every user-facing doc (source-boundaries 245-249, source-packs 138-160, cli-reference 70, architecture 56, CHANGELOG 51-55) correctly says "library" — but a design-of-record marked "Complete" that describes the wrong architecture will mislead maintainers. Update the spec body (or add a "shipped as: library import" note pointing at the Stage 232 plan).

## Nits
- **`InstagramSourceSettings.instaloader_path` is dead config** (`src/fashion_radar/models/source.py:69`). It was the subprocess design's binary-path override; the library collector never reads it. With `extra="forbid"` a user can set it and it silently no-ops. It is undocumented (the source-packs example omits it) and harmless, but it is the second residue of the subprocess→library pivot — consider removing it.
- **`_classify` labels generic/connection failures `instaloader_unavailable`** (`collectors/instagram.py:86-93`) even though instaloader is installed (a transient network error reads as "unavailable"). Fail-closed either way and explicitly tested, and consistent with Xiaohongshu's `*_unavailable` catch-all, so this is only a diagnostics-label nuance.
- Spec wording `items_seen=<discovered count>` is stale; `len(bounded)` is the correct choice for a lazy iterator you intentionally don't drain. Folds into the spec-refresh above.

## Summary
Phase 3 ships coherent, opt-in, use-at-your-own-risk Instagram acquisition. The boundary was already opt-in (Stage 221); the collector wraps the user-installed `instaloader` *library* as an optional import (fail-closed to `instaloader_unavailable` when absent), reuses the user's saved session via `load_session_from_config` with **no Fashion-Radar password/session handling**, forces the lazy hashtag/profile iterator inside a try so auth/connection errors become `login_required`/skip, and bounds output with `itertools.islice(..., max_posts_per_run)` (≤200). It registers in `_default_collectors()` and is in both runner enrichment-skip guards, mirroring the Xiaohongshu pattern. No new Python dependency (optional import; `uv.lock`/`pyproject.toml` untouched) and no schema change. User-facing docs are consistent and carry the opt-in / use-at-your-own-risk / first-live-run field-verification caveat, the boundary text is locked by a new docs-guard test (`test_source_boundaries_docs_describe_instagram_opt_in`), and X/YouTube remain deferred to Phases 4-5 (no `SourceType` members; CHANGELOG and spec scope confirm). Title rendering follows the established title-raw / summary-via-`report_safe_snippet` pattern, so it adds no new markdown exposure beyond the separately-tracked Stage 210 snippet-hygiene work. The only blemishes are the stale internal design spec and the dead `instaloader_path` field — both documentation/cleanup debt, neither release-blocking. Ship it, with a follow-up to reconcile the spec.
