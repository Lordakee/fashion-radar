# HTML Report Code Review
**Verdict:** REQUEST_CHANGES

## Critical
- **No HTML escaping of external/untrusted data (stored XSS + broken markup).** Every interpolated value in `html_report.py` is written raw via f-strings — `entity_name`, `label`, `phrase`, `title`, `summary`, `source_name`, `source_type`, `last_error_message`, `run.status`, `brief.summary`, `section.title`, `reason_codes`, and `item.source_url`. This data originates from RSS feeds and scraped web pages (the collectors), i.e. untrusted external input. Two concrete failures:
  - A title/summary containing `<script>…</script>` (or any tag) is executed when the operator opens the report in a browser — stored XSS against the local file, able to read the page and make outbound requests. Note `collectors/rss.py` calls `html.unescape()` on ingest, so raw `<`/`>`/`&` are actively decoded *into* the stored text before they ever reach this renderer, making malformed/hostile content the expected case, not the edge case.
  - Even benign fashion content breaks rendering: `Dolce & Gabbana`, a quote in a brand name, or `Margiela <show recap>` corrupts the HTML (`&` entity, swallowed `<…>`, attribute breakout).
  - `href='{item.source_url}'` is single-quoted and unescaped: a `'` in the URL breaks out of the attribute, and a `javascript:` scheme URL executes on click.
  - Fix: route all text through `html.escape()` (the stdlib `html` module is already used in the codebase), and for `source_url` escape with `quote=True` plus validate the scheme is `http`/`https` before emitting the anchor. This also restores parity with the project's existing "report-safe" posture (Stage 210 collapsed these same error/snippet fields for the Markdown path; the HTML path reintroduces the raw values). I'd add a regression test feeding `<script>`, `&`, `'`, and a `javascript:` URL.

## Important
- **HTML artifact is written but never returned or surfaced.** `write_daily_report_files` still returns `tuple[Path, Path]` (md, json) and the `.html` path is dropped on the floor. Callers (`cli.py:1428`, `cli.py:2203`) and the Stage-11 digest packaging only ever see the two returned paths, so the new file won't appear in digests/community-handoff manifests. If the manifest/handoff verification enumerates or checksums files in `reports/`, an un-manifested `.html` could either be silently omitted or trip an "unexpected file" check. I did not trace the manifest code here — please confirm the handoff/manifest path either includes or tolerates the new artifact, and decide whether the path should be returned.

## Nits
- **`.gitignore` inconsistency.** Only `reports/**/*.html` was added; the existing md/json/eml/txt entries carry *both* `reports/*.ext` and `reports/**/*.ext`. The generated file is top-level (`reports/fashion-radar-{date}.html`), and `reports/**/*.html` does match it (git's `/**/`  matches zero directories — `a/**/b` matches `a/b`), so it is ignored. I confirmed this by gitignore semantics, not execution (`git check-ignore` was permission-gated). Add `reports/*.html` for consistency with the established convention.
- **`_render_health` appends "..." unconditionally.** `source.last_error_message[:80] + "..."` adds an ellipsis even when the message is ≤80 chars (a 10-char error renders `short error...`). Compare `_render_representative_items`, which correctly guards with `len(...) > 120`. Apply the same length guard.
- **Missing parameter type hints.** Helpers are untyped (`_render_brief(brief)`, `_render_signals(entities)`, …) while the rest of the codebase is fully typed. Ruff's selected rules (E,F,I,UP,B) won't flag it, but annotate with `DailyBrief`, `list[EntityReport]`, etc. for consistency.
- **`if not brief` is effectively dead.** `DailyReport.brief` has `default_factory=empty_daily_brief` and a pydantic model is always truthy, so the branch never triggers; the empty-report path still renders the empty-brief summary. Harmless, just slightly misleading.
- **Per-file E501 ignore is broad.** Disabling E501 for the whole file also silences it for the Python logic, not just the CSS template. Acceptable and was called out as expected; `# noqa: E501` on template lines only would be more surgical.

## Summary
The feature is well-structured and the CSS template is clean. Verification of your six points: (1) valid HTML — confirmed (DOCTYPE, `lang`, charset/viewport/title, four sections, footer, all CSS braces correctly doubled for `str.format`); (2) empty fallbacks — confirmed for signals/candidates/health and the empty brief; (3) integration — confirmed at `workflows.py:70-71`, though the path isn't returned (see Important); (4) `.gitignore` — functionally covered, but stylistically inconsistent (nit); (5) E501 per-file-ignore — present and reasonable; (6) no external deps — confirmed, fully self-contained with inline `<style>`, system font stack, and `rel="noopener"` on outbound links. The blocker is the complete absence of output escaping: external feed/scrape content flows unescaped into an HTML file a human opens in a browser, which is both an XSS vector and a guaranteed markup-corruption bug for ordinary fashion text. Escape all text and validate/escape the URL, add a special-character regression test, and this is good to go.
