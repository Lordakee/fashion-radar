- `Critical:` `docs/superpowers/plans/2026-06-12-stage-9-manual-signal-import-plan.md`, Task 5 "CLI Command", lines 848-864 / requirements lines 879-882: the planned real-import flow creates the SQLite engine and calls `initialize_schema(engine)` before `import_manual_signal_rows()` validates the file. In this codebase, `create_sqlite_engine(default_database_path(data_dir))` creates `data_dir`, and `initialize_schema(engine)` opens a write path and creates schema tables. That violates the stated Stage 9 architecture requirement: "Validate all rows before opening any write path." Required change: restructure the plan so the CLI validates the import file before creating the engine, initializing the schema, or creating `data_dir`. For example: parse `--imported-at`; call `load_manual_signal_rows()` first; if validation fails, exit without DB/data-dir creation; only then create engine, initialize schema, and persist the already-validated rows. Add a CLI test proving an invalid non-dry-run import does not create `data_dir` or `fashion-radar.sqlite`.

- `Important:` `docs/superpowers/plans/2026-06-12-stage-9-manual-signal-import-plan.md`, Task 4 "Import Workflow": `import_manual_signal_rows(engine, ...)` validates before using `ItemRepository`, but because the API requires an already-created `Engine`, it does not itself guarantee "before opening any write path" unless callers follow the right sequence. The CLI plan currently does not. Recommended change: split the API into validation and persistence helpers, e.g. `load_manual_signal_rows()` plus `store_manual_signal_rows(engine, rows, ...)`, or make the command explicitly validate first and pass prevalidated rows to the write function. This makes the architecture harder to accidentally violate.
