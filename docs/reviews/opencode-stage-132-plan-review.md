# Stage 132 Plan Review

## Critical findings
None. The approach is sound: it reuses the existing `find_secret_findings()` (preserving patterns, redaction, and line numbering), reuses the already-computed `git ls-files --others --exclude-standard` untracked set, leaves `safe_repo_path()` untouched (so resolved-path containment stays intact as defense-in-depth), and adds a narrowly-scoped symlink skip. Nothing in the design weakens the forbidden-path policy, docs-verification surfaces, dependency/lockfile state, archive checker, or any product behavior. Git does not dereference symlinks at commit time, so skipping untracked symlinks does not open a real leak path, and the ignored target in the symlink test is correctly excluded by `--exclude-standard` before any content scan.

## Important findings
1. **Robustness of `find_secret_findings()` against arbitrary untracked content.** Tracked content is curated; untracked content can be binary, huge (fresh build artifacts not yet gitignored), or non-UTF-8. The plan must confirm the function already skips binary/oversized/undecodable files without raising, otherwise the new untracked scan can introduce new crash modes and false positives that the tracked scan never hit. This is the single biggest behavioral risk of widening the scope from tracked → untracked.
2. **"Empty stderr" in the symlink test is brittle.** `test_untracked_symlink_target_is_not_scanned_for_secret` asserts release hygiene success *and* empty stderr. The symlink skip must be completely silent (no info/warn log about skipped symlinks), or this test will fail. Either guarantee silence or relax to "no secret finding and no raw token in stderr."
3. **Intermediate-directory symlinks.** `(repo_root / normalized).is_symlink()` only inspects the final path component. A path like `linkdir/notes.md` where `linkdir` is a symlink to `/etc` is not caught by the new skip; it falls through to `safe_repo_path()`, whose `.resolve()` + `relative_to()` will reject it. Confirm that rejection path inside `find_secret_findings()` is handled gracefully (skip/reported finding) and does not raise, since that code path is now reachable via untracked input.
4. **Self-flagging of Stage/review artifacts by the live gate.** The verification step runs `check_release_hygiene.py --repo-root .`, which will now content-scan every unignored untracked file in the working tree, including this stage's review notes. Any token-shaped string (`ghp_…`) or copy-pasteable `extraheader`/auth-header push command in the review docs will trip the gate. The plan acknowledges this; ensure all such artifacts use obvious placeholders that do *not* match the GitHub-token regex and contain no literal `http(s)://<token>@`/`Authorization:` push snippets.
5. **Provenance label in the untracked finding.** The content test expects a "line-numbered untracked finding." Confirm the `"untracked"` label passed to `find_secret_findings()` actually surfaces in the rendered finding text so operators can distinguish tracked vs untracked leaks; otherwise the label parameter is dead and the test's intent is only half-verified.
6. **Single computation of the untracked set.** Plan states `untracked_paths` "already comes from" the existing `git ls-files --others --exclude-standard` call. Confirm it is reused (not recomputed) so ordering/filtering stays identical to the forbidden-path scan that already consumes untracked paths — a second invocation could diverge on sort order or dirty-tree state and produce flaky findings.

## Minor findings
1. **TOCTOU on `is_symlink()`.** A filesystem race between the `is_symlink()` check and the read is theoretically possible, but this is a local developer hygiene gate, not a security boundary, so it is acceptable.
2. **Performance.** Scanning all untracked files line-by-line can be slow if large unignored artifacts are present pre-commit; acceptable, but a sane size cap (already implied by Important #1) mitigates it.
3. **Path-normalization assumptions.** `(repo_root / normalized)` with forward-slash repo-relative paths is fine on this Linux repo; worth a one-line note that normalization matches the tracked-path branch exactly so the two scans are symmetric.
4. **Test fixture hygiene.** The "valid `ghp_` fixture" must satisfy only the GitHub-token pattern and must not also match the persistent extraheader/authorization patterns, or the test could pass/fail for the wrong reason and the extraheader scan step could misreport.
5. **No dedup needed.** Because symlinks are skipped and tracked/untracked sets are disjoint, there is no risk of double-reporting the same physical file; the plan correctly does not introduce dedup logic.

## Final statement
No Critical blockers. The Important findings above are confirmations/robustness guards rather than design defects; they should be verified or tightened before release (especially untracked-content robustness in `find_secret_findings`, silent symlink skip, graceful handling of intermediate-symlink rejection, and ensuring Stage/review artifacts contain no token-shaped or copy-pasteable auth-header strings). With those confirmed, there are no Critical or Important blockers to release.
