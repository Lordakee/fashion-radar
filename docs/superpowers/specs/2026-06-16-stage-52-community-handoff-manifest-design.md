# Stage 52 Community Handoff Manifest Design

## Objective

Add a local, print-only `community-handoff-manifest` command that external
user-controlled community/social tools can read before producing directory
handoff files for Fashion Radar.

## Background

Fashion Radar already has two complementary producer aids:

- `community-signal-profile` prints the field-level producer contract.
- `community-handoff-workflow` prints copyable commands for validating,
  previewing, importing, and reviewing one local handoff directory.

External tools still need one directory-oriented manifest that combines the
target directory, filename/pattern expectations, producer contract pointers,
and local command workflow in a stable machine-readable shape. This stage fills
that gap without adding any source acquisition, scraping, browser automation,
platform login, cookies/sessions, platform APIs, scheduling, monitoring, or
connector code.

## Technical Stack

- Python 3.11+
- Typer for CLI registration and options.
- Pydantic models with `extra="forbid"` for deterministic JSON contracts.
- Existing `community_signal_profile` constants and
  `build_community_handoff_workflow()` for shared contract data and command
  generation.
- Pytest and current CLI/documentation/package tests.
- No new runtime or development dependencies.

## Proposed Command

```bash
fashion-radar community-handoff-manifest DIRECTORY \
  --input-format csv \
  --pattern "*.csv" \
  --config-dir "$PWD/configs" \
  --data-dir "$PWD/data" \
  --as-of "$AS_OF" \
  --source-name "Community Tool Export" \
  --format json
```

Table output is the default. JSON output is intended for external tools that
want to inspect the contract before writing files.

## Manifest Shape

The manifest will be produced by a new
`fashion_radar.community_handoff_manifest` module:

- `contract_version`: `community-handoff-manifest/v1`
- `execution_mode`: `print_only`
- `directory`, `input_format`, `pattern`, `as_of`, `config_dir`, `data_dir`,
  `source_name`
- `producer_profile_command`: `fashion-radar community-signal-profile --format json`
- `producer_contract_version`, `supported_input_formats`, `prohibited_fields`,
  `field_notes`, `field_rules`, and `unsupported_capabilities` copied from
  `build_community_signal_profile()` so a manifest consumer can avoid a second
  CLI call for core field rules
- `suggested_filename`: `community-signals.csv` or `community-signals.json`
- `matched_file_rule`: downstream lint/preview/import commands treat
  direct-child regular files matching the supplied pattern as handoff files
- `manifest_storage_note`: if a caller saves the manifest, keep it outside the
  matched handoff directory or use a filename excluded by the handoff pattern;
  especially do not save a `.json` manifest inside the handoff directory when
  using `--pattern "*.json"`
- `schema_path`, `example_paths`, `csv_header`, `required_fields`,
  `optional_fields`, `json_envelopes`
- `workflow`: nested `CommunityHandoffWorkflow` object generated by the existing
  workflow builder
- `boundaries`: concise local-only, print-only boundaries

The JSON key order will be tested to make this safe for downstream tools.

## Behavior

- The command prints the manifest only.
- It parses `--as-of` with the same timestamp handling as
  `community-handoff-workflow`.
- It does not inspect the supplied directory path.
- It does not read handoff files, validate files, import rows, open SQLite,
  create config/data/report/dashboard artifacts, execute subprocesses, fetch
  URLs, log in, store cookies, call platform APIs, monitor communities, schedule
  work, or add platform connectors.
- It may print supplied directory/config/data paths because the output is a
  producer-facing local contract and command manifest.
- If an external tool saves the manifest JSON, it should not save that file in
  the same directory/pattern that `community-signal-lint-dir`,
  `community-candidates-dir`, or `import-signals-dir` will treat as signal
  files. For JSON batches using `--pattern "*.json"`, the saved manifest must
  not be a `.json` file inside the handoff directory.

## Implementation Plan

1. Add `src/fashion_radar/community_handoff_manifest.py` with Pydantic models,
   a builder, and a table renderer.
2. Register `community-handoff-manifest` in `src/fashion_radar/cli.py` using the
   same directory/input-format/pattern/config/data/as-of/source-name/format
   option style as `community-handoff-workflow`.
3. Add focused unit tests for deterministic model fields, filename selection,
   nested workflow reuse, quoting, invalid timestamps, and table sanitization.
4. Add CLI tests for help, JSON output, table output, invalid `--as-of`, and no
   directory/artifact side effects.
5. Update README, community signal import docs, source boundaries, CLI
   reference, GitHub upload checklist, and package/doc tests so the public
   command surface stays synchronized.
6. Run the full verification set before commit and GitHub upload.

## Testing Strategy

- Unit tests for the new module.
- CLI tests with `CliRunner`.
- Documentation tests that ensure the command appears in CLI reference and the
  installed-wheel help loop.
- Existing package archive tests to ensure source distributions include the new
  source module through the existing package layout.
- Full test suite, Ruff, lock check, mirror-based sync check, release hygiene,
  package archive check, and first-run smokes before upload.

## Out Of Scope

- Platform search for Instagram, TikTok, X, Xiaohongshu, or other services.
- Browser automation, account login, cookie/session storage, scraping/crawling,
  source acquisition, source ranking, demand proof, platform coverage, media
  download, scheduling, watching, or monitoring.
- Any compliance-review workflow.
